# Confectioner: Telegram-бот конструктора тортов

Telegram-бот для кондитера: пользователь выбирает готовый торт, подбирает изделие из каталога или собирает индивидуальный заказ, а администратор ведёт заказы, управляет каталогом и получает отзывы.

## Важно для разработки

При добавлении, изменении или удалении функционала обязательно обновляйте этот `README.md`: пользовательские сценарии, админские сценарии, модели БД, скрипты, настройки и архитектурные решения должны оставаться актуальными.

## Технологии

- Python 3.11+
- aiogram 3.x
- SQLAlchemy 2.0 async
- PostgreSQL через `DATABASE_URL`
- FSM: `MemoryStorage`
- Планировщик: APScheduler

## Запуск

1. Создайте `.env`:

```env
BOT_TOKEN=telegram-bot-token
DATABASE_URL=postgresql+asyncpg://user:password@host:5432/db
ADMIN_TELEGRAM_ID=123456789
BASE_PRICE=2500
SESSION_TIMEOUT_MIN=30
SESSION_CHECK_INTERVAL_MIN=5
RUN_DB_MIGRATIONS=1
```

2. Установите зависимости:

```bash
python -m venv .venv
.venv/bin/pip install -r requirements.txt
```

3. Инициализируйте схему:

```bash
.venv/bin/python scripts/init_db.py
```

4. Наполните базовыми тестовыми данными:

```bash
.venv/bin/python scripts/seed.py
```

5. При необходимости расширьте описание продуктов до `TEXT`, добавьте поля оформления заказа и наполните каталог тортов:

```bash
.venv/bin/python scripts/alter_product_description_text.py
.venv/bin/python scripts/alter_checkout_and_account_fields.py
.venv/bin/python scripts/alter_logging_analytics.py
.venv/bin/python scripts/add_catalog_cakes.py
```

6. Запустите бота:

```bash
.venv/bin/python bot.py
```

## Запуск в Docker

PostgreSQL в контейнере не поднимается. Бот подключается к уже существующей базе по `DATABASE_URL`.

Файлы Docker:

- `Dockerfile` — образ бота;
- `docker-compose.yml` — один сервис `bot`;
- `docker-entrypoint.sh` — применяет идемпотентные обновления схемы перед запуском;
- `.dockerignore` — исключает `.env`, `.venv`, логи и кэш из Docker build context.

### Локальная проверка Docker

1. Создайте `.env`:

```env
BOT_TOKEN=telegram-bot-token
DATABASE_URL=postgresql+asyncpg://user:password@postgres-host:5432/db
ADMIN_TELEGRAM_ID=123456789
BASE_PRICE=2500
SESSION_TIMEOUT_MIN=30
SESSION_CHECK_INTERVAL_MIN=5
RUN_DB_MIGRATIONS=1
```

2. Соберите образ:

```bash
docker compose build
```

3. Запустите бота:

```bash
docker compose up -d
```

4. Посмотрите логи:

```bash
docker compose logs -f bot
```

5. Остановите:

```bash
docker compose down
```

Если схема базы уже актуальна и не нужно запускать скрипты при каждом старте, поставьте:

```env
RUN_DB_MIGRATIONS=0
```

### Развёртывание на сервере

1. Установите Docker и Docker Compose plugin на сервере.

Для Ubuntu:

```bash
sudo apt update
sudo apt install -y docker.io docker-compose-plugin
sudo systemctl enable --now docker
```

2. Скопируйте проект на сервер:

```bash
git clone <repo-url> confectioner
cd confectioner
```

Если репозитория на сервере нет, можно загрузить папку проекта любым удобным способом.

3. Создайте `.env` на сервере:

```bash
cp .env.example .env
nano .env
```

Минимально нужны:

```env
BOT_TOKEN=telegram-bot-token
DATABASE_URL=postgresql+asyncpg://user:password@postgres-host:5432/db
ADMIN_TELEGRAM_ID=123456789
BASE_PRICE=2500
SESSION_TIMEOUT_MIN=30
SESSION_CHECK_INTERVAL_MIN=5
RUN_DB_MIGRATIONS=1
```

Важно: `postgres-host` должен быть доступен с сервера, где запущен контейнер. Если PostgreSQL находится на другом сервере или в managed-сервисе, проверьте firewall, security groups и разрешение подключения для пользователя базы.

4. Соберите и запустите:

```bash
docker compose up -d --build
```

При первом старте entrypoint выполнит:

```bash
python scripts/init_db.py
python scripts/alter_product_description_text.py
python scripts/alter_checkout_and_account_fields.py
python scripts/alter_logging_analytics.py
```

Эти операции безопасны для повторного запуска: таблицы создаются без удаления данных, а изменения схемы используют `IF NOT EXISTS`.

5. Проверьте состояние:

```bash
docker compose ps
docker compose logs -f bot
```

6. Если нужно один раз наполнить базу стартовыми компонентами или каталогом тортов:

```bash
docker compose exec bot python scripts/seed.py
docker compose exec bot python scripts/add_catalog_cakes.py
```

Не запускайте `seed.py` повторно без необходимости, если в базе уже есть реальные данные.

7. Обновление на сервере:

```bash
git pull
docker compose up -d --build
docker compose logs -f bot
```

8. Перезапуск:

```bash
docker compose restart bot
```

9. Остановка:

```bash
docker compose down
```

Логи бота сохраняются в папке `logs/` на сервере через volume `./logs:/app/logs`.

## Архитектура

Точка входа: `bot.py`.

Бот создаёт `Bot`, `Dispatcher(storage=MemoryStorage())`, подключает `RelayMiddleware`, регистрирует роутеры и запускает polling. Перед polling вызывается `delete_webhook(drop_pending_updates=True)`.

При старте вызывается `utils/logging_config.py:setup_logging()`. Логи пишутся в консоль и в `logs/bot.log` с ротацией.

Регистрация роутеров находится в `handlers/__init__.py`. Порядок важен:

1. `handlers/admin/products.py`
2. `handlers/admin/media.py`
3. `handlers/admin/orders.py`
4. `handlers/admin/monitoring.py`
5. `handlers/user/catalog.py`
6. `handlers/user/checkout.py`
7. `handlers/user/account.py`
8. `handlers/user/templates.py`
9. `handlers/user/funnel.py`
10. `handlers/user/order.py`
11. `handlers/user/review.py`
12. `handlers/common.py`
13. `handlers/admin/relay.py`

Админские роутеры подключаются раньше пользовательских, а общий роутер навигации подключается ближе к концу.

## Структура проекта

```text
bot.py                      # запуск бота
config.py                   # настройки из .env
db/
  base.py                   # engine, async_session, Base
  models.py                 # SQLAlchemy-модели
  repositories/             # слой доступа к данным
handlers/

  user/                     # пользовательские сценарии
  admin/                    # админ-панель

scripts/                    # инициализация и сервисные операции с БД
```

## Модель данных

Основные таблицы описаны в `db/models.py`.

### users

Пользователи Telegram. Поле `is_admin` выставляется по `ADMIN_TELEGRAM_ID`.

### user_sessions

Черновики пользовательской воронки. Используются для восстановления незавершённой сборки и мониторинга брошенных сессий.

Scheduler проверяет эти записи по `SESSION_TIMEOUT_MIN`. Если пользователь не завершил сборку, бот отправляет пользователю напоминание с кнопками “Продолжить” и “Начать заново”, а админу отправляет уведомление с кнопками “Написать клиенту” и “Открыть сессию”.

### funnel_events

События воронки для аналитики:

- `funnel_started`;
- `step_selected`;
- `date_reached`;
- `confirmation_shown`;
- `order_created`;
- `funnel_abandoned`.

Поля:

- `user_id`;
- `session_id`, может быть `NULL`;
- `event_type`;
- `step`;
- `payload`;
- `created_at`.

Связь с `user_sessions` настроена как `ON DELETE SET NULL`, чтобы завершённые заказы могли удалять черновик сессии, не теряя события аналитики.

### components

Компоненты каталога и конструктора:

- `occasion`
- `persons`
- `shape`
- `filling`
- `decor`

Компоненты имеют `image_url`, `short_description`, `price_delta`, `weight_grams`, `sort_order`, `is_active`.

`short_description` используется для коротких подсказок в конструкторе, например “универсальный вариант для детей”. `weight_grams` особенно важен для компонентов типа `persons`: по нему бот показывает примерный вес заказа.

### products

Готовые изделия/шаблоны. Используются в:

- “Популярные заказы”
- “Подобрать из готовых вариантов”

Важные поля:

- `name`
- `description`
- `price`
- `image_url`
- `is_template`
- `status`

`description` имеет тип `TEXT`, потому что описания тортов могут быть длинными.

### product_components

Связь готового продукта с компонентами. Эта таблица критична для фасетного подбора по каталогу.

Продукт без `product_components` может отображаться как готовый шаблон, но не будет участвовать в подборе из готовых вариантов.

### orders и order_components

Заказы и состав индивидуального заказа. Для готового продукта `product_id` заполнен. Для кастомного заказа `product_id = NULL`, а состав хранится в `order_components` и `description`.

Дополнительные поля заказа:

- `delivery_type`: `delivery` или `pickup`;
- `delivery_address`;
- `delivery_comment`;
- `delivery_time_from`;
- `delivery_time_to`;
- `customer_comment`;
- `admin_comment`;
- `result_image_url`.

Эти поля используются в личном кабинете пользователя, админской карточке заказа и уведомлениях.

### calendar_days

Календарь доступности дат. Поля:

- `date`;
- `status`: `available`, `busy`, `blocked`;
- `comment`.

Перед оформлением заказа бот проверяет выбранную дату. Если дата занята или заблокирована, пользователь получает ближайшие доступные даты.

### reviews

Текстовые отзывы после закрытия заказа. Фото отзыва не сохраняется в БД, а пересылается админу в Telegram.

### payments и component_conflicts

Таблицы присутствуют в схеме, но полноценная оплата и ограничения совместимости сейчас не являются основным активным сценарием.

## Пользовательские сценарии

Главное меню:

- Популярные заказы
- Создать самому
- Подобрать из готовых вариантов
- Личный кабинет
- Справка

### Популярные заказы

Файл: `handlers/user/templates.py`.

Показывает активные `products`, где `is_template=True`. Если у продукта есть `image_url`, бот отправляет карточку с фото. После выбора пользователь проходит общий блок оформления:

1. Дата доставки/самовывоза.
2. Проверка даты по `calendar_days`.
3. Способ получения: доставка или самовывоз.
4. Адрес доставки, если выбрана доставка.
5. Временной интервал.
6. Итоговый экран и подтверждение.

### Создать самому

Файл: `handlers/user/funnel.py`.

Свободный конструктор по компонентам. Шаги:

1. Ориентировочный бюджет.
2. Повод.
3. Количество персон.
4. Форма.
5. Оформление: можно выбрать до трёх элементов.
6. Дата и пожелания.
7. Способ получения.
8. Временной интервал.
9. Подтверждение.

На каждом шаге показываются активные компоненты нужного типа. Цена считается как `BASE_PRICE + сумма price_delta выбранных компонентов`.

При наличии `image_url` у компонентов бот показывает карточки с фото. Иначе показывает компактный список кнопок.

Шаг “Начинка” в свободном конструкторе не используется. Начинки остаются частью готовых продуктов и каталожного подбора.

### Подобрать из готовых вариантов

Файлы:

- `handlers/user/catalog.py`
- `services/catalog_service.py`

Это фасетный подбор по `products` и `product_components`.

Логика:

- пользователь выбирает форму;
- после формы бот показывает список подходящих изделий;
- при нажатии на изделие бот показывает изображение, описание, состав и цену;
- пользователь может согласиться с выбором или вернуться назад;
- после согласия пользователь проходит общий блок оформления: дата, получение, время, подтверждение;
- если найден один продукт, бот может сразу показать карточку;
- если точного результата нет, бот предлагает подобрать заново или перейти в индивидуальный заказ.

На первом шаге выбора формы кнопки показывают количество вариантов. На остальных этапах количество вариантов на кнопках не показывается.

### Подтверждение заказа

Файл: `handlers/user/order.py`.

Поддерживает:

- заказ готового продукта;
- заказ из каталога;
- кастомный заказ.
- повтор существующего заказа из личного кабинета.

После подтверждения создаётся запись в `orders`, администратору отправляется уведомление.

### Оформление заказа

Файл: `handlers/user/checkout.py`.

Это общий блок для популярных заказов, каталога, свободного конструктора и повторения заказа.

Сценарий:

1. Пользователь вводит дату в формате `ДД.ММ.ГГГГ`.
2. Бот проверяет `calendar_days`.
3. Если дата занята, бот показывает ближайшие доступные даты.
4. Пользователь выбирает способ получения:
   - доставка;
   - самовывоз.
5. Для доставки бот спрашивает адрес и комментарий курьеру.
6. Пользователь выбирает временной интервал:
   - `10:00-12:00`;
   - `12:00-14:00`;
   - `14:00-16:00`;
   - `16:00-18:00`.
7. На итоговом экране показываются состав, дата, получение, время, адрес, примерный вес и цена.

### Личный кабинет

Файл: `handlers/user/account.py`.

Раздел главного меню “Личный кабинет” содержит:

- “Мои заказы”;
- “Написать кондитеру”.

“Мои заказы” показывает активные и завершённые заказы пользователя. Пользователь может открыть заказ и увидеть:

- состав торта;
- цену;
- желаемую дату;
- способ получения и время;
- статус;
- таймлайн статуса;
- комментарий администратора;
- свой комментарий к заказу;
- фото результата, если оно есть;
- кнопку “Повторить заказ”.

Повтор заказа:

1. Пользователь открывает старый заказ.
2. Нажимает “Повторить заказ”.
3. Бот показывает состав, примерный вес и прошлую цену.
4. Пользователь подтверждает повтор или меняет дату.
5. Далее используется общий checkout: дата, получение, время, подтверждение.

Пользователь может отменить заказ, изменить дату или изменить комментарий только в статусах `created` и `confirmed`. После перехода заказа в работу бот предлагает связаться с администратором.

### Отзыв после закрытия заказа

Файл: `handlers/user/review.py`.

Когда админ переводит заказ в статус `closed`, пользователь получает кнопку “Оставить отзыв”. Сценарий:

1. Пользователь пишет текст.
2. Может прикрепить фото или пропустить.
3. Текст сохраняется в `reviews`.
4. Отзыв и фото, если оно есть, отправляются админу.

## Админские сценарии

Админ определяется через `ADMIN_TELEGRAM_ID`.

После `/start` админ получает постоянную reply-кнопку “Открыть панель”.

### Заказы

Файл: `handlers/admin/orders.py`.

Админ может:

- открыть список заказов;
- смотреть открытые и закрытые заказы;
- открыть карточку заказа;
- менять статус.
- написать клиенту из карточки;
- отправить массовые напоминания клиентам с заказами на завтра.

Статусы:

- `created`
- `confirmed`
- `in_progress`
- `ready`
- `paid`
- `closed`
- `cancelled`

Кнопки действий сформулированы глаголами: “Подтвердить”, “В работу”, “Отметить готовым”, “Отметить оплаченным”, “Закрыть”, “Отменить”.

Список заказов показывает номер, статус и краткое описание.

Карточка заказа показывает:

- клиента;
- дату;
- цену;
- статус;
- способ получения;
- временной интервал;
- адрес доставки;
- состав;
- описание/пожелания;
- комментарий администратора.

В панели также есть пункты “Найти заказ” и “Создать заказ”. Сейчас они заведены как точки входа интерфейса с описанием ожидаемого сценария; полноценный пошаговый поиск и ручное создание требуют отдельной реализации FSM.

### Мониторинг воронки

Файл: `handlers/admin/monitoring.py`.

Показывает активные пользовательские сессии из `user_sessions`: пользователь, текущий шаг, сколько минут прошло с обновления, признак брошенной сессии.

Из уведомления о брошенной сессии админ может:

- написать клиенту через relay;
- открыть карточку сессии и посмотреть текущий шаг/черновик.

### Статистика

Файл: `handlers/admin/monitoring.py`.

Кнопка “Статистика” в админской панели показывает:

- сегодня: количество заказов, выручку, средний чек;
- неделю: количество заказов, отмены, закрытые заказы;
- популярные начинки за 7 дней;
- воронку за 7 дней: старт конструктора, дошли до даты, увидели итог, создали заказ, брошенные сессии.

### Создание продукта

Файл: `handlers/admin/products.py`.

Админский мастер создания продукта:

1. Изображение.
2. Название.
3. Описание.
4. Цена.
5. Состав/ингредиенты.
6. Привязка компонентов каталога: форма, начинка, оформление, персоны, повод.
7. Подтверждение.

Созданный продукт получает:

- `is_template=True`
- `status=active`
- связи в `product_components`

Если компонентные связи не задать, продукт не будет участвовать в фасетном подборе.

### Редактирование каталога

Файл: `handlers/admin/products.py`.

Админ может редактировать:

- продукты: фото, название, описание, цену, компоненты;
- компоненты: фото, название, надбавку, активность.

### Загрузка изображений

Файл: `handlers/admin/media.py`.

Админ может отправить изображение боту и получить `file_id`. Также можно сразу сохранить изображение:

- `product 3`
- `component 7`

Значение сохраняется в `image_url`. В проекте поддерживаются Telegram `file_id`, URL и локальные пути.

### Relay-диалог

Файлы:

- `handlers/admin/relay.py`
- `middlewares/relay_mw.py`
- `services/relay_service.py`

Админ может начать диалог с пользователем. Пока relay активен, текстовые сообщения пользователя перехватываются middleware и отправляются админу, не ломая текущую FSM-сессию пользователя.

## Сервисы

### funnel_service.py

Логика свободного конструктора:

- шаги конструктора;
- заголовки шагов;
- получение активных компонентов;
- расчёт цены;
- сбор текстового описания состава.

Компоненты кэшируются на короткое время для ускорения.

### catalog_service.py

Логика фасетного подбора:

- порядок шагов каталога;
- выбор кандидатов по уже выбранным компонентам;
- расчёт доступных вариантов, которые не ведут в тупик;
- сбор краткого состава продукта.

### order_service.py

Создание заказов:

- кастомный заказ;
- заказ готового продукта;
- повтор заказа;
- смена статуса.

### checkout.py

Пользовательский общий сценарий оформления:

- ввод и проверка даты;
- ближайшие доступные даты;
- доставка или самовывоз;
- адрес доставки;
- временной интервал;
- итоговый экран с примерным весом и ценой.

### notify_service.py

Уведомления:

- админу о новом заказе;
- пользователю о смене статуса;
- предложение оставить отзыв после закрытия заказа.

### scheduler.py

Планировщик проверяет брошенные сессии:

- отправляет пользователю напоминание продолжить или начать заново;
- уведомляет администратора;
- пишет событие `funnel_abandoned`;
- логирует ошибки scheduler и ошибки отправки сообщений.

### logging_config.py

Настройка логирования:

- консоль;
- `logs/bot.log`;
- ротация по 5 МБ, 5 файлов.

Логируются:

- создание заказа;
- смена статуса;
- ошибки отправки сообщений;
- ошибки удаления сообщений;
- ошибки scheduler;
- ошибки relay;
- события воронки.

## Репозитории

Репозитории находятся в `db/repositories/`.

- `users.py`: создание/получение пользователей.
- `products.py`: список шаблонов, каталог, создание и редактирование продуктов.
- `components.py`: список, получение и редактирование компонентов.
- `orders.py`: создание заказов, выборка с отношениями, статусы.
- `calendar.py`: проверка доступности дат и подбор ближайших свободных дней.
- `funnel_events.py`: запись событий воронки.
- `analytics.py`: сводка заказов, популярных компонентов и воронки.
- `sessions.py`: черновики сессий и мониторинг.
- `reviews.py`: сохранение отзывов.

## Изображения

Поля `image_url` могут содержать:

- Telegram `file_id`;
- HTTP/HTTPS URL;
- локальный путь к файлу.

Функция `utils/message.py:image_input` превращает локальные пути в `FSInputFile`, а Telegram `file_id` и URL оставляет строкой.

## Удаление сообщений и скорость

Файл: `utils/message.py`.

Бот хранит ID последних сообщений в FSM и удаляет их при переходах. Для пользовательских сценариев удаление часто запускается в фоне, чтобы интерфейс отвечал быстрее. В админке многие переходы используют редактирование текущего сообщения вместо удаления и отправки нового.

## Как добавить новый торт вручную

Через админ-панель:

1. Нажать “Создать продукт”.
2. Загрузить изображение или пропустить.
3. Ввести название.
4. Ввести описание.
5. Ввести цену.
6. Ввести состав.
7. Выбрать компоненты каталога.
8. Подтвердить создание.

Чтобы торт участвовал в “Подобрать из готовых вариантов”, у него должны быть связи в `product_components`.

## Как добавить новый торт скриптом

1. Добавить недостающие компоненты в `BASE_COMPONENTS` или как `filling` у торта.
2. Добавить запись в `CAKES`.
3. Запустить:

```bash
.venv/bin/python scripts/add_catalog_cakes.py
```

## Как проверить основные сценарии

### Пользователь

1. `/start`.
2. “Популярные заказы” → выбрать торт → подтвердить.
3. “Создать самому” → выбрать бюджет, повод, персон, форму, до трёх элементов оформления → дату → получение → время → подтвердить.
4. “Подобрать из готовых вариантов” → выбрать форму → выбрать изделие → посмотреть карточку → согласиться → дату → получение → время → подтвердить.
5. “Личный кабинет” → “Мои заказы” → открыть заказ → повторить заказ.
6. После закрытия заказа админом оставить отзыв.

### Админ

1. `/start`.
2. “Открыть панель”.
3. “Статистика”.
4. “Открыть мониторинг”.
5. “Открыть заказы”.
6. Открыть заказ и поменять статус.
7. Из карточки заказа написать клиенту.
8. Отправить напоминания клиентам на завтра.
9. “Создать продукт”.
10. “Редактировать каталог”.
11. Отправить фото с подписью `product <id>` или `component <id>`.

## Известные ограничения

- Полноценная оплата пока не включена в основной пользовательский сценарий.
- `component_conflicts` есть в схеме, но ограничения совместимости сейчас не применяются в основной логике.
- Фото отзывов пересылаются админу, но не сохраняются отдельным полем в БД.
- Свободный конструктор не проверяет совместимость компонентов.
- Фасетный подбор работает только по продуктам, у которых заполнены `product_components`.
- Поиск заказа и ручное создание заказа добавлены в админское меню как точки входа, но полноценные пошаговые сценарии ещё не реализованы.
